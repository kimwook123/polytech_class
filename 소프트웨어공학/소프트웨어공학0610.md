# 소프트웨어공학

소프트웨어
: 개발 과정에서 생성되는 모든 산출물
  각 단계에서 만들어지는 문서, 사용자 메뉴얼

소프트웨어 개발 모델

1. 주먹구구식 모델(Build-and-fix)
: 공시적인 가이드라인이나 프로세스가 없는 개발 방식
- 요구분석명세서, 설계 없음
- 관리 및 유지보수 힘듦(문서x)
- 지속적 수정 가능성이 높음

---

1. 선형 순차적 모델
: 표준 프로세스를 통해 소프트웨어를 순차적으로 개발하는 방식
- 하향식 개발 방식
- 문서 중심의 모델
- 요구사항 변화가 거의 없고, 선형적인 작업 진행 개발에 유용
- 선형 순차적 모델- 폭포수 모델
계획→분석→설계→구현→테스트→유지보수
- 관리 용이
- 체계적 문서화
- 단계적 절차로 인한 대기 발생 및 요구사항 검증에 제한적임
- 특정 단계에서 발생한 오류가 다음 단계로 이월됨
- 중간 평가에 제한적임

---

1. 진화적 프로세스 모델
- 프로토타입 모델
- 가시적 결과물
- 점진적 요구사항 정의
- 사용자의 적극적 참여
- 인력/비용 산정 문제
- 나선형 모델
- 요구분석→위험분석→프로토타입→사용자평가→보완 및 수정
- 반복적 요구사항 반영을 통한 사전에 위험을 인식하고, 사용자 만족이 높아짐
- 기간이 장기화되고, 위험 관리 전문가가 필요하다.

---

1. 단계적 개발 모델
: 개발과 사용을 병행하는 과정 반복 진행
- 점층적 개발 방법: 개발 범위 증가
- 중요한 부분부터 개발 후, 개발 범위를 늘려가는 방식
- 초기 비용 감소
- 서브 시스템 간 연관성 고려 및 통합이 어려움
- 반복적 개발 방법: 품질 증가
- 시스템 전체를 1차적으로 개발 및 인도한 후에, 각 서브시스템의 기능과 성능 변경 및 보완

---

1. 통합 프로세스 모델
- 반복적 개발 방법의 일종이다.
- 빈번한 요구사항 변경에 대해 유연한 대응이 가능하다.
- 도입→구체화→구축→전이

---

1. 애자일 프로세스 모델
- 고객 요구에 민첩하게 대응하고, 주어진 문제를 주기적으로 풀어나가는 방법론
- 추가 요구사항 수용 가능
- 2주~2개월 간격으로 동작 가능한 소프트웨어 제공
- 담당자-개발자 간 주기적인 의견 공유
- 스크럼
: 팀의 개선과 프로젝트 관리에 중점을 둔 경험적 관리 기법
1) 제품 기능 목록 작성
2) 스프린트 계획 회의
3) 스프린트 수행
4) 스프린트 개발 완료
5) 스프린트 검토 및 다음 스프린트 계획

---

UML
: 시스템의 상호작용, 시스템 전체 구조, 컴포넌트 간의 관계 등을 시각적으로 볼 수 있게 나타낸 도면

유스케이스 다이어그램
- 식별→검증→명칭 지정→목록화

- 식별
- 사용자가 요구하는 기능을 사용자의 업무에서 도출
- DB 등록/수정/삭제/조회 기능 확인
- 유스케이스와 액터와의 상호작용 확인
- 검증
- 수작업 제외
- 액터가 원하는 최종 결과만을 나타내는지 확인
- 유스케이스 내 이벤트 흐름 전체를 액터가 사용하는지 확인
- 명칭 지정
- 최종 목적을 나타내는 이름으로 지정

---

클래스 다이어그램: 클래스 정의, 클래스 간 관계/역할을 표현한 것

---

순차 다이어그램
: 실행 시, 객체들의 상호 동작을 메시지 순서 기반으로 표현
구성요소
- 객체, 객체생명선, 메시지, 동기/비동기 메시지, 시나리오 정의, 시나리오 분할, 다이어그램과 시나리오의 1:1 대응

---

통신 다이어그램
: 메시지를 이용해 객체 간의 상호작용 표현
구성요소
- 객체, 메시지, 링크

---

활동 다이어그램
: 객체의 행위를 표현/행위가 수행되는 동안의 활동을 표현

- 분기: 조건에 의해 두 가지 경로로 나뉘는 위치 (<< >> 기호 )
- 병합: 분기되어 수행하던 활동이 합쳐지는 위치 - 마름모 기호
- 동기화 막대: 여러 활동 병합
- 신호: 활동 사이의 거래를 제어하기 위한 신호
- 구획면: 해당 레인을 책임지는 객체 표현

---

상태 다이어그램
: 객체의 상태 변화를 표현

구성요소

- 상태: 모서리가 둥근 사각형으로 작성
- 전이: 화살표로 작성
- 이벤트: << >> 기호로 작성

---

컴포넌트 다이어그램
구성: 인터페이스, 의존 관계, 의존 관계 정의

---

배치 다이어그램
: 하드웨어 자원을 명시적으로 정의
  시스템의 물리적 요소의 모델링 및 노드 간의 관계 표현

---

계 획

- 문제 정의
- 타당성 분석
- 경제적, 기술적, 법적 타당성
- 개발 비용 산정
- 하향식 산정 기법
: 경험을 바탕으로 회의를 통해 산정하는 비 과학적 기법
- 상향식 산정 기법
: 세부 작업을 단위별로 비용을 산정한 후에 전체 비용을 합산하는 방법
- 수학적 산정 기법
1. COCOMO 방법?
→ LOC를 추정한 후, 사전에 정의된 수식에 대입하여 개발 인건비를 예측하는 것

2. COCOMO 2 방법
→ 개발 초기 단계에서 정확한 LOC 예측이 어려운 것을 보완한 방법
애플리케이션 합성 모델, 초기 설계 모델, 구조 설계 이후 모델 등

3. 기능 점수 산정 방법
→ 기능의 수를 기반으로 복잡도를 판단하여 비용을 산정하는 방법
